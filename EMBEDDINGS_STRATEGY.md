# Embeddings Strategy

## Вердикт

Embeddings стоит внедрять.

Но внедрять их нужно:

- отдельно от сырого текста;
- параллельно текущей аналитике;
- как semantic layer поверх существующей модели данных, а не как замену текущих полей и логики.

Ключевой принцип: сырой текст отзывов остается основным источником правды, а embeddings используются для семантического поиска, retrieval, кластеризации и дальнейшей аналитики.

## Что хранить

Обязательно хранить:

- сырой текст отзыва;
- embeddings этого отзыва.

Нельзя заменять одно другим. Нужно хранить оба слоя данных.

## Почему сырой текст обязателен

Сырой текст нужен:

- для отображения отзыва в интерфейсе;
- для аудита и ручной проверки;
- для отладки рекомендаций и чат-ответов;
- для повторной обработки новыми моделями;
- для генерации рекомендаций на основе реальных формулировок клиентов;
- для безопасной смены провайдера embeddings без потери данных.

## Зачем нужны embeddings

Embeddings дают:

- семантический поиск по смыслу, а не по точным словам;
- поиск похожих отзывов;
- группировку близких проблем;
- retrieval для LLM: передавать в модель не все отзывы, а только наиболее релевантные;
- базу для кластеров, тем, трендов и интеллектуальной навигации по отзывам.

## Как хранить правильно

Лучше хранить embeddings в отдельной таблице.

Не рекомендую складывать вектор прямо в `feedback_responses`, если задача воспринимается как долгосрочная архитектура.

Причины:

- можно перевычислять embeddings без изменения основной таблицы;
- можно менять модель векторизации без болезненных миграций доменной таблицы;
- можно хранить служебные поля индексации;
- можно переиндексировать данные пакетно;
- проще versioning и поддержка нескольких провайдеров или моделей.

## Рекомендуемая схема

### Таблица отзывов

`feedback_responses` остается доменной таблицей с бизнес-данными:

- `id`
- `user_id`
- `feedback_text`
- `created_at`
- score-поля:
  - `score_result`
  - `score_explanation`
  - `score_comfort`
  - `score_booking`
  - `score_recommendation`

### Таблица embeddings

Новая таблица, например `feedback_embeddings`:

- `id`
- `feedback_id`
- `user_id`
- `provider`
- `model`
- `dimension`
- `embedding`
- `text_hash`
- `status`
- `error_message`
- `created_at`
- `updated_at`

### Что означают служебные поля

- `provider`: например `yandex`
- `model`: конкретная модель embeddings
- `dimension`: размерность вектора
- `text_hash`: позволяет понять, изменился ли текст и нужно ли перевычислять embedding
- `status`: например `pending`, `ready`, `failed`
- `error_message`: причина сбоя индексации

## Почему отдельная таблица лучше навсегда

Если в будущем:

- поменяется модель embeddings;
- изменится размерность;
- появится повторная индексация;
- захочется хранить несколько векторов на один отзыв;
- понадобится фоновая обработка и статусы;

то отдельная таблица выдерживает это без переделки основной бизнес-модели.

Это правильнее для системы, которую потом не хочется переделывать.

## Как embeddings должны жить в системе

Embeddings не должны становиться новой основной логикой приложения.

Они должны быть дополнительным интеллектуальным слоем:

- `feedback_responses` отвечает за исходные данные;
- `feedback_embeddings` отвечает за смысловое представление;
- рекомендации, поиск, чат и аналитика используют оба слоя вместе.

## Где embeddings реально полезны в этом проекте

### 1. Семантический поиск по сырым отзывам

Это один из самых полезных и быстрых по ценности сценариев.

Примеры:

- найти отзывы про ожидание, даже если люди пишут `ждала`, `задержали`, `не предупредили`;
- найти отзывы про болезненность процедуры, даже если формулировки разные;
- найти отзывы по теме коммуникации, без ручных тегов.

### 2. Поиск похожих отзывов

Можно показывать:

- похожие жалобы;
- похожие позитивные кейсы;
- повторяющиеся паттерны.

Это полезно и для аналитики, и для рекомендаций.

### 3. Retrieval для генерации рекомендаций

Это особенно полезно.

Вместо того чтобы отправлять в LLM все отзывы подряд, можно:

- взять релевантные отзывы по диапазону;
- отфильтровать их embeddings-поиском;
- выбрать 3-10 самых содержательных;
- передать в генерацию только их.

Это снижает шум и повышает качество рекомендаций.

### 4. Умный чат по отзывам

Embeddings хорошо подходят для сценария:

1. пользователь задает вопрос;
2. система ищет наиболее близкие отзывы;
3. найденные отзывы подаются в LLM как контекст;
4. модель отвечает на основе реальных данных.

### 5. Автоматическое выявление тем

Это уже второй этап, но он имеет смысл:

- ожидание;
- коммуникация;
- комфорт;
- цена;
- объяснения;
- стерильность;
- результат процедуры.

Сначала embeddings, потом тематическая аналитика поверх них.

## Что не стоит делать первым этапом

Я не рекомендую сразу строить на embeddings всю бизнес-логику.

Не стоит начинать с:

- сложной ежедневной кластеризации как основы продукта;
- попытки заменить текущие score-поля;
- сравнения периодов только по векторам;
- очень тяжелой аналитики до появления достаточного массива данных.

Причина простая: это добавит сложность раньше, чем появится реальная отдача.

## Что embeddings не заменяют

Embeddings не заменяют явные оценки:

- `Результат`
- `Объяснения`
- `Комфорт`
- `Запись`
- `Рекомендация`

Эти score-поля и embeddings решают разные задачи:

- score-поля дают явную структуру и численную аналитику;
- embeddings дают смысловую близость и гибкий retrieval.

Правильная система использует оба подхода одновременно.

## Практическая рекомендация по этапам

### Этап 1

Добавить отдельную таблицу `feedback_embeddings` и хранение embeddings рядом с отзывами.

### Этап 2

Делать embeddings асинхронно после создания или обновления отзыва:

- отзыв записывается сразу;
- embedding считается отдельно;
- при сбое отзыв не теряется.

### Этап 3

Использовать embeddings для:

- semantic search;
- похожих отзывов;
- retrieval в рекомендациях;
- retrieval в чат-боте.

### Этап 4

Только после этого добавлять:

- кластеризацию;
- автоматические темы;
- динамические semantic filters;
- продвинутую аналитику по трендам.

## Как это встроить в текущий стек

Для текущей архитектуры правильнее всего:

- оставить `feedback_responses` как есть;
- добавить `feedback_embeddings`;
- использовать PostgreSQL + pgvector;
- генерировать embeddings через Yandex AI Studio;
- не смешивать индексацию отзывов и генерацию рекомендаций в одну синхронную операцию.

## Как должна работать запись нового отзыва

Правильный pipeline:

1. сохранить отзыв в `feedback_responses`;
2. поставить задачу на индексацию;
3. получить embedding через API;
4. сохранить embedding в `feedback_embeddings`;
5. отметить статус готовности.

Если embeddings временно не посчитались, отзыв все равно должен жить в системе.

## Как должна работать генерация рекомендаций

Я бы видел это так:

1. пользователь выбирает диапазон;
2. система берет отзывы за период;
3. при необходимости делает semantic retrieval внутри этого массива или поверх всей истории;
4. выбирает наиболее релевантные отзывы;
5. передает их в LLM;
6. сохраняет рекомендацию вместе с метаданными.

То есть embeddings усиливают рекомендации, но не заменяют текущую доменную логику.

## Архитектурный принцип

Правильная модель для этого проекта:

- сырой текст: источник правды;
- embeddings: слой семантики;
- score-поля: слой структурной аналитики;
- LLM: слой интерпретации и генерации.

Это дает устойчивую архитектуру без лишней магии.

## Итог

Если делать "один раз и правильно", то решение такое:

- embeddings внедрять;
- хранить параллельно сырому тексту;
- хранить в отдельной таблице;
- использовать сначала для semantic search и retrieval;
- не заменять ими текущие оценки и базовую аналитику;
- не делать их центром всей системы на первом этапе.

Итоговый вывод:

Embeddings для этого проекта нужны.
Но нужны как отдельная подсистема смыслового поиска и retrieval, а не как замена существующей логики рекомендаций и рейтингов.
