# Mbooking: план внедрения подписки через Яндекс Пэй

Документ фиксирует внедрение оплаты подписки 500 ₽ / 30 дней через Яндекс Пэй в текущую архитектуру Mbooking.

---

## 1) Что берем из Яндекс Пэй для вашего кейса

Для SaaS-подписки полезны сценарии:
- онлайн-оплата на сайте;
- оплата по ссылке как fallback канал;
- web-сценарий checkout для быстрого запуска.

Цель интеграции:
- пользователь оплачивает 500 ₽;
- backend получает подтверждение оплаты;
- доступ к premium-фичам (`lessons`, `recommendations`) открывается на 30 дней.

---

## 2) Целевая бизнес-логика доступа

- До оплаты: базовые функции доступны, premium закрыт.
- После оплаты: premium доступ на 30 дней.
- По окончании периода: premium блокируется до продления.
- Доступ не ограничивается специализацией мастера: все уроки доступны любому активному подписчику.

---

## 3) Архитектура backend (без привязки к одному провайдеру)

Чтобы не зависеть от одного PSP, сделать провайдерный слой:

- `BillingProvider` interface:
  - `createCheckout(...)`
  - `parseWebhook(...)`
  - `isSuccessEvent(...)`
  - `extractPaymentIdentity(...)`

- Реализация:
  - `YandexPayProvider implements BillingProvider`

Это позволит позже параллельно держать ЮKassa как второй канал.

---

## 4) Модели данных Prisma

Использовать общий billing-контур:
- `OwnerSubscription`
- `PaymentTransaction`

Ключевые поля:
- `provider` = `yandex_pay`
- `providerPaymentId`
- `status` (pending/succeeded/canceled/failed)
- `currentPeriodStart/currentPeriodEnd` в подписке

---

## 5) Эндпоинты backend для Яндекс Пэй

Создать в `backend/src/modules/billing`:

- `POST /v1/billing/yandex-pay/checkout`
  - создает платеж;
  - возвращает URL/токен для перехода к оплате.

- `POST /v1/billing/yandex-pay/webhook` (public)
  - принимает события провайдера;
  - идемпотентно фиксирует успешный платеж;
  - активирует/продлевает подписку.

- `GET /v1/billing/subscription`
  - возвращает статус доступа для UI/SSR guard.

---

## 6) Активация подписки (30 дней)

Правило при успешной оплате:
- если подписки нет или она истекла:
  - `start = now`
  - `end = now + 30 days`
- если подписка активна:
  - `end = currentPeriodEnd + 30 days`

Статус подписки: `active`.

## 8) Надежность и безопасность

- Подписку активирует только webhook.
- Проверять сумму (`500 RUB`) и идентификатор платежа.
- Использовать идемпотентность при создании и обработке платежа.
- Ограничить частоту вызова checkout endpoint.
- Логировать все платежные события с корреляционным id.

---

## 9) Пошаговый план внедрения

1. Подключить договор/аккаунт Яндекс Пэй для бизнеса.
2. Получить тестовые и боевые реквизиты, настроить callback/webhook URL.
3. Реализовать `YandexPayProvider` + checkout endpoint.
4. Реализовать webhook endpoint + идемпотентную обработку.
5. Подключить `SubscriptionAccessService` к `recommendations` и `lessons`.
6. Добавить UI-страницы billing/paywall.
7. Провести sandbox тесты:
   - successful payment;
   - canceled payment;
   - duplicate webhook;
   - timeout/повтор.
8. Выпустить в прод и включить мониторинг.

---

## 10) Риски

- Могут отличаться требования API/событий от привычной схемы checkout.
- Нужно заранее проверить, насколько прозрачен серверный сценарий подтверждения оплаты.
- При изменениях тарифа/условий понадобится адаптерный слой (поэтому обязателен `BillingProvider`).

---

## 11) KPI запуска

- конверсия в оплату со страницы paywall;
- доля успешных checkout;
- среднее время до активации подписки;
- доля пользователей с активной подпиской через 30 дней;
- usage premium-функций после оплаты.
