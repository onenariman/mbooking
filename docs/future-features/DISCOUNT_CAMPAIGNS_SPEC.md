# DISCOUNT CAMPAIGNS SPEC

> Спецификация для будущей страницы скидок, акций и массовых предложений.
> Документ описывает отдельный продуктовый контур поверх уже реализованных персональных скидок.

---

## 0. Зачем нужен отдельный раздел

Текущая система уже умеет:
- назначать скидку конкретному клиенту
- привязывать скидку к конкретной услуге
- применять скидку при записи и списывать ее только после завершения визита

Но для маркетинга этого недостаточно.

Нужен отдельный раздел, в котором мастер сможет:
- создать акцию на конкретную услугу
- назначить скидку всем клиентам сразу
- ограничить срок действия акции, например на 7 дней
- видеть историю акций
- позже отправлять push-уведомления по этим акциям
- получать подсказки, каким клиентам стоит предложить скидку

Вывод: это должна быть не просто кнопка в карточке услуги, а отдельная страница уровня `Скидки / Акции / Кампании`.

---

## 1. Главная идея

### 1.1 Что хотим получить

Пример сценария:
- появилась новая услуга
- мастер заходит в раздел скидок
- выбирает услугу
- ставит скидку `20%`
- задает срок действия `7 дней`
- нажимает `Разослать всем клиентам`

Система:
- создает кампанию
- назначает персональные скидки всем подходящим клиентам
- проставляет `expires_at`
- сохраняет историю
- позже может отправить push в клиентский кабинет

### 1.2 Почему это отдельная сущность, а не просто массовая вставка скидок

Потому что потом нужно видеть:
- кто создал акцию
- на какую услугу
- какой процент
- с какого по какое время действует
- сколько скидок создано
- сколько клиентов уже использовали акцию
- активна акция или уже завершена

То есть нужен не просто набор `client_discounts`, а еще и сущность `campaign`.

## 2. Новый раздел в интерфейсе

### 2.1 Новая страница

Нужна отдельная страница:
- `Скидки`

Или более явно:
- `Акции`
- `Кампании`

Рекомендуемое название для admin UI:
- `Скидки`

Внутри страницы можно сделать 3 блока:
- `Персональные скидки`
- `Кампании`
- `Рекомендации`

### 2.2 Что должно быть на странице

#### Блок 1. Персональные скидки
- список уже назначенных скидок
- фильтр по клиенту
- фильтр по услуге
- фильтр по статусу: активна / использована / истекла
- ручное создание персональной скидки

#### Блок 2. Кампании
- список массовых акций
- кнопка `Новая акция`
- история созданных кампаний
- статус кампании: черновик / активна / завершена / отменена
- количество клиентов, которым назначена скидка
- количество использований

#### Блок 3. Рекомендации
- список клиентов, которым стоит предложить скидку
- причина рекомендации
- быстрые действия:
  - `назначить скидку`
  - `создать кампанию`
  - `отложить`

## 3. Продуктовые сценарии

### 3.1 Массовая скидка на новую услугу

Сценарий:
- мастер выбирает новую услугу
- задает скидку `10-30%`
- задает срок действия, например `7 дней`
- задает текст уведомления
- выбирает аудиторию `все клиенты`
- запускает кампанию

Результат:
- клиентам создаются скидки на эту услугу
- скидки видны в клиентском кабинете
- позже по ним можно отправить push

### 3.2 Точечная реактивация

Сценарий:
- система показывает, кто давно не приходил
- мастер выбирает клиентов из списка
- назначает скидку только им

### 3.3 Полуавтоматические рекомендации

Сценарий:
- система сама предлагает: `10 клиентов не были 45 дней`
- мастер подтверждает предложение
- система создает скидки и кампанию

### 3.4 AI-assisted рекомендации

Сценарий:
- AI анализирует историю визитов
- AI предлагает, кому и на какую услугу дать скидку
- мастер подтверждает руками

Важно:
- AI не должен автоматически рассылать скидки без подтверждения мастера

---

## 4. Что рекомендую сделать сначала

### 4.1 Сначала без AI

Я рекомендую сначала сделать rule-based механику без AI.

Причина:
- она прозрачнее
- дешевле в реализации
- проще тестируется
- не требует объяснять, почему AI выбрал именно этих клиентов
- дает ту же бизнес-пользу уже на первом этапе

Примеры правил без AI:
- клиент не приходил `30+` дней
- клиент не приходил `60+` дней
- клиент был только один раз
- клиент оставлял отзыв, но еще не вернулся
- клиент пользовался одной услугой, но не пробовал новую

### 4.2 Потом добавить AI как слой поверх правил

AI лучше добавлять как помощника:
- анализировать сегменты
- предлагать формулировку акции
- предлагать размер скидки
- предлагать аудиторию
- предлагать причину рекомендации

Но финальное действие должно оставаться за мастером.

Итоговая рекомендация:
- `V1`: кампании и rule-based рекомендации без AI
- `V2`: AI-подсказки и AI-объяснения

## 5. Предлагаемая структура данных

### 5.1 Новая таблица `discount_campaigns`

```sql
discount_campaigns
- id uuid pk
- user_id uuid not null
- title text not null
- service_id uuid null
- service_name_snapshot text null
- discount_percent integer not null
- expires_at timestamptz not null
- audience_type text not null
- message_text text null
- status text not null
- created_at timestamptz not null
- created_by uuid not null
```

Рекомендованные значения:
- `audience_type`:
  - `all_clients`
  - `inactive_clients`
  - `selected_clients`
- `status`:
  - `draft`
  - `active`
  - `completed`
  - `cancelled`

### 5.2 Расширение `client_discounts`

Нужно добавить:
- `campaign_id uuid null`
- расширить `source_type`

Было:
- `feedback`
- `manual`

Станет:
- `feedback`
- `manual`
- `campaign`

### 5.3 Возможно понадобится таблица получателей кампании

Если захотим хорошую аналитику:

```sql
discount_campaign_recipients
- id uuid pk
- campaign_id uuid not null
- client_id uuid null
- client_phone text not null
- discount_id uuid null
- delivery_status text null
- delivered_at timestamptz null
- used_at timestamptz null
- created_at timestamptz not null
```

Это не обязательно для первой версии, но полезно для истории и статистики.

## 6. Правила кампаний

### 6.1 Кампания на услугу

Кампания должна быть привязана к услуге:
- `service_id`
- `service_name_snapshot`

Это позволит:
- показывать в клиентском кабинете понятный текст
- применять скидку только на нужную услугу
- делать точечные уведомления

### 6.2 Срок действия обязателен

Для массовых акций срок действия должен быть обязательным.

Пример:
- `expires_at = now() + interval '7 days'`

### 6.3 Без дублей

Если у клиента уже есть активная скидка на эту же услугу, нужно не плодить дубли.

Рекомендуемое правило для V1:
- не создавать вторую активную скидку на ту же услугу
- помечать клиента как `skipped`

### 6.4 Только подтвержденный запуск

Перед массовой рассылкой нужен экран подтверждения:
- услуга
- размер скидки
- срок действия
- сколько клиентов попадет в кампанию

## 7. Рекомендации без AI

### 7.1 Что можно сделать сразу

На странице `Скидки` нужен блок `Рекомендации`.

Он может строиться простыми SQL-правилами:
- не были `30` дней
- не были `45` дней
- не были `60` дней
- были только один раз
- давно не посещали конкретную услугу

### 7.2 Примеры полезных сегментов

#### Сегмент A. Давно не приходили
- клиент не был `45+ дней`

#### Сегмент B. Пропали после первого визита
- у клиента только один завершенный визит
- с него прошло `30+ дней`

#### Сегмент C. Не пробовали новую услугу
- клиент раньше ходил на похожие услуги
- новая услуга появилась недавно
- можно предложить акционную цену

#### Сегмент D. Имеют активную скидку, но еще не использовали
- можно напомнить перед истечением срока

### 7.3 Почему этого уже достаточно

Потому что это решает главную задачу:
- система сама подсказывает, кому сейчас стоит сделать предложение

И это можно реализовать надежно, быстро и без лишней сложности.

## 8. AI-блок

### 8.1 Где AI реально полезен

AI может:
- анализировать историю клиентов
- группировать похожие паттерны
- предлагать идею акции
- предлагать текст пуша
- объяснять, почему именно этим клиентам стоит отправить скидку

### 8.2 Где AI не должен быть первым слоем

Не стоит сразу строить критичную механику только на AI:
- выбор аудитории
- автоматическая массовая выдача скидок
- автоматическая отправка push без подтверждения

### 8.3 Правильная роль AI

AI должен быть:
- помощником
- генератором гипотез
- объясняющим слоем

Но не единственным источником истины.

### 8.4 Пример AI-подсказки

Пример карточки рекомендации:
- `12 клиентов не приходили более 52 дней`
- `8 из них раньше пользовались уходом за лицом`
- `Рекомендация: предложить 15% на новую чистку лица на 7 дней`

Кнопки:
- `Создать кампанию`
- `Изменить параметры`
- `Скрыть`

---

## 9. Как это будет связано с клиентским кабинетом

Когда появится клиентский кабинет:
- клиент увидит активные скидки по услугам
- клиент увидит срок действия
- клиент сможет получить push:
  - `Вам назначена скидка 20% на новую услугу "Чистка лица"`
  - `Скидка истекает через 2 дня`

То есть эта страница скидок напрямую связана с будущим client portal и push-уведомлениями.

## 10. Предлагаемый порядок внедрения

### Этап 1. Основа без AI
- новая страница `Скидки`
- таблица `discount_campaigns`
- массовое создание скидок на услугу
- срок действия кампании
- аудитория `все клиенты`
- история кампаний

### Этап 2. Сегментация без AI
- блок `давно не приходили`
- блок `только один визит`
- блок `есть активная скидка, но не использована`
- быстрые действия по созданию скидок

### Этап 3. Push для кампаний
- уведомление клиенту о новой скидке
- напоминание о скором истечении скидки

### Этап 4. AI-помощник
- AI-рекомендации по сегментам
- AI-подсказка размера скидки
- AI-генерация текста акции

## 11. Итоговое решение

Решение правильное:
- делать это через новую страницу
- выделять кампании в отдельную сущность
- строить V1 без AI
- добавить AI позже как помощника

Итоговая позиция:
- функция обязательно нужна
- ее место на отдельной странице скидок
- AI нужен, но не первым слоем
