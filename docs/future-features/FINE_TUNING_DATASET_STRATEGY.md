# Fine-Tuning Dataset Strategy

## Зачем этот документ

Этот документ фиксирует будущую идею:

- если в проекте появится fine-tuning;
- если захочется строить собственный датасет знаний;
- если потребуется обучать модель под стиль, логику и формат ответов;

то делать это нужно через чистый и управляемый pipeline данных, а не через случайный сбор чужих материалов.

## Базовый вердикт

Если делать fine-tuning в этом проекте, то лучше строить его на:

- собственных данных;
- лицензированных данных;
- данных с явным разрешением;
- собственных переработанных структурированных записях.

Не стоит строить основной training corpus на:

- сырых YouTube-транскриптах;
- полных текстах книг без прав;
- массово собранных чужих материалах из открытого интернета.

## Главный принцип

Нужно различать:

- источник идей;
- training data.

Источник идей может использоваться для исследования.
Но training data для fine-tuning лучше собирать как отдельный оригинальный актив проекта.

## Что считается хорошей стратегией

Хорошая стратегия:

1. изучать внешние материалы;
2. извлекать принципы, паттерны, ошибки, подходы;
3. формировать собственные обучающие примеры;
4. дообучать модель уже на собственном датасете.

## Что считается плохой стратегией

Плохая стратегия:

- брать массово чужие тексты;
- слегка чистить их;
- пересжимать через ИИ;
- использовать это как основной training corpus;
- надеяться, что этого никто не заметит.

Это плохая долгосрочная база для продукта.

## Что нужно собирать вместо сырых чужих материалов

Нужно собирать собственные структурированные примеры.

### Подходящие форматы

- `вопрос -> ответ`
- `ситуация -> анализ -> рекомендация`
- `ошибка -> почему это ошибка -> как правильно`
- `жалоба клиента -> причина -> корректный ответ`
- `слабый ответ -> сильный ответ`
- `сырой отзыв -> выводы -> приоритет действий`
- `тема -> сигналы -> риски -> действия`

## Как использовать внешние материалы безопаснее

Внешние материалы лучше использовать не как датасет, а как:

- исследовательскую базу;
- источник идей;
- источник общих паттернов;
- ориентир по качеству и глубине объяснения.

После этого нужно создавать новый собственный текст.

## Принцип переработки

Правильный путь:

- не хранить большой массив чужих текстов как корпус;
- не делать fine-tuning на почти прямых summary чужих материалов;
- создавать свои независимые записи, написанные под нужды продукта.

## Как должен выглядеть будущий dataset

Ниже пример того, что лучше подходит для fine-tuning, чем чужие длинные тексты.

### Формат 1. Instruction tuning

- `instruction`
- `context`
- `response`

Пример:

- `instruction`: проанализируй негативный отзыв и предложи 3 шага улучшения
- `context`: кратко структурированные сигналы из отзыва
- `response`: конкретный управленческий ответ

### Формат 2. Style tuning

Нужен, если модель надо донастроить под конкретную манеру:

- ясная структура;
- короткие практические советы;
- ответ без воды;
- ответ с акцентом на действия.

### Формат 3. Domain tuning

Если задача доменная:

- отзывы;
- сервис;
- клиентский опыт;
- запись;
- комфорт;
- объяснения мастера;
- рекомендации по улучшению.

Тогда лучше собирать именно кейсы из своего домена, а не общий мотивационный контент.

## Самая сильная идея для этого проекта

Для этого проекта fine-tuning, если и делать, то лучше не на "бизнес-коучах вообще", а на собственном прикладном наборе:

- отзывы клиентов;
- их структурная интерпретация;
- хорошие управленческие рекомендации;
- хорошие ответы на вопросы по сервису;
- хорошие форматы суммаризации;
- шаблоны объяснений и рекомендаций.

То есть дообучение должно усиливать именно продукт, а не быть общей энциклопедией чужих советов.

## Откуда брать данные

Наиболее безопасные источники:

- свои тексты;
- свои заметки;
- свои методички;
- собственные базы кейсов;
- контент сотрудников/экспертов компании;
- тексты по договору или лицензии;
- public domain;
- материалы с явным разрешением на такое использование.

## Более рискованные источники

- чужие YouTube-транскрипты;
- книги без лицензии;
- платные курсы;
- приватные Telegram/Discord/Slack материалы;
- форумы и закрытые сообщества;
- чужие PDF и методички без прав.

## Практический safe workflow

### Этап 1. Исследование

Изучаются:

- книги;
- ролики;
- статьи;
- лекции;
- кейсы;
- собственные рабочие материалы.

Но это еще не training data.

### Этап 2. Извлечение идей

Из материалов вынимаются:

- общие принципы;
- часто повторяющиеся паттерны;
- полезные структуры решений;
- ошибки и анти-паттерны;
- типовые сценарии.

### Этап 3. Создание собственного датасета

На основе идей формируются уже новые записи:

- своими словами;
- в едином формате;
- под задачу продукта;
- без привязки к формулировкам конкретного автора.

### Этап 4. Ревизия качества

Каждая запись проверяется:

- понятна ли она;
- оригинальна ли она по форме;
- не утащила ли она уникальную подачу исходного автора;
- соответствует ли она реальной продуктовой задаче.

### Этап 5. Fine-tuning

Только после этого датасет используется для настройки модели.

## Что можно сделать до fine-tuning

До полноценного fine-tuning можно получить большую пользу и без него:

- промпт-инжиниринг;
- шаблоны ответов;
- retrieval;
- embeddings;
- knowledge base на своих материалах;
- few-shot примеры.

Это дешевле и безопаснее на старте.

## Когда fine-tuning реально оправдан

Fine-tuning оправдан, если:

- нужен устойчивый стиль ответов;
- нужен стабильный формат вывода;
- нужно меньше зависеть от длинных промптов;
- уже накоплен собственный качественный датасет;
- есть повторяющийся класс задач.

Если качественного собственного датасета нет, то fine-tuning часто преждевременен.

## Какой dataset лучше всего подошел бы этому проекту

### Потенциальные типы записей

- `отзыв -> краткие выводы -> 3 действия`
- `группа отзывов -> основные проблемы -> план улучшений`
- `негативный отзыв -> корректный ответ клиенту`
- `позитивный отзыв -> что усиливать в сервисе`
- `смешанный отзыв -> что улучшить без потери сильных сторон`
- `сигналы по оценкам -> интерпретация -> управленческая рекомендация`

## Минимальный draft полей датасета

### Вариант A

- `instruction`
- `input`
- `output`

### Вариант B

- `scenario`
- `signals`
- `target_style`
- `response`

### Вариант C

- `raw_feedback`
- `normalized_signals`
- `recommended_actions`

## Чего не делать

- не делать training corpus из спорного массива внешних текстов;
- не надеяться на "никто не узнает";
- не хранить проблемные исходники без необходимости;
- не путать исследовательские заметки и production dataset;
- не обучать модель на плохо очищенном и случайном корпусе.

## Рекомендуемая последовательность

### Этап 1

Собрать внутренний стиль ответов и шаблоны.

### Этап 2

Собрать вручную небольшой качественный датасет.

### Этап 3

Проверить, хватает ли prompt + retrieval без fine-tuning.

### Этап 4

Если нет, запускать fine-tuning уже на собственном корпусе.

## Итог

Если в будущем в проекте появится fine-tuning, то правильная стратегия такая:

- внешние материалы использовать как исследование;
- training data строить как собственный актив;
- делать упор на оригинальные структурированные записи;
- fine-tuning привязывать к задачам продукта, а не к чужим контент-корпусам.

Главная цель:

не просто "накормить модель большим числом текстов", а построить чистый, полезный и защищаемый dataset.
