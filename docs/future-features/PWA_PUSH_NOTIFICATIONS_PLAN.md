# PWA Push Notifications Plan

## Цель

Реализовать push-уведомления для `Home Screen web app`, чтобы приложение в браузере ощущалось ближе к нативному приложению.

Под этим здесь понимается:

- пользователь устанавливает сайт на главный экран;
- разрешает уведомления;
- получает системные push-уведомления;
- по нажатию открывается нужный экран приложения.

Это особенно важно для iPhone и iPad: полноценный web push там нужно проектировать именно под `Home Screen web app`, а не под обычную вкладку браузера.

## Текущая ситуация в проекте

Сейчас в проекте нет готовой PWA-базы:

- нет `manifest`;
- нет `service worker`;
- нет регистрации `navigator.serviceWorker`;
- нет логики `PushManager`;
- нет таблиц хранения подписок;
- нет backend-слоя для отправки web push.

Это значит, что внедрение нужно делать поэтапно.

## Архитектурный вердикт

Push-уведомления для этого проекта делать стоит.

Но делать их нужно не как случайный фронтенд-хак, а как отдельную подсистему:

- PWA-слой;
- подписки устройств;
- настройки уведомлений;
- отправка уведомлений с сервера;
- обработка кликов и деактивация невалидных подписок.

## Как это должно работать в продукте

### Пользовательский сценарий

1. Пользователь открывает приложение.
2. Устанавливает его на главный экран.
3. Заходит уже как в отдельное приложение.
4. Нажимает кнопку включения уведомлений.
5. Разрешает уведомления.
6. Устройство получает push-подписку.
7. Подписка сохраняется в базе.
8. Бэкенд отправляет push по нужным событиям.
9. По нажатию на уведомление открывается конкретный экран приложения.

## Ограничения платформ

### Общее

Для web push нужны:

- `HTTPS`;
- `Service Worker`;
- явное разрешение пользователя;
- корректная подписка через `PushManager`.

### iPhone / iPad

Для Apple нужно учитывать:

- push для web app нормально рассматриваем именно в сценарии `Add to Home Screen`;
- сайт в обычной вкладке Safari не должен быть основной целевой моделью для push UX;
- опыт нужно строить вокруг установленного `Home Screen web app`.

## Что нужно внедрить

## 1. PWA-основа

Нужно добавить:

- `manifest.webmanifest` или `app/manifest.ts`;
- иконки приложения;
- `apple-touch-icon`;
- `theme_color`;
- `background_color`;
- `display: "standalone"`;
- `start_url`;
- `scope`.

### Что это даст

- приложение можно будет установить;
- оно будет открываться как отдельное приложение;
- появится корректная база для web push.

## 2. Service Worker

Нужен отдельный service worker, который будет:

- принимать событие `push`;
- показывать уведомление через `showNotification`;
- обрабатывать `notificationclick`;
- открывать или фокусировать приложение на нужном маршруте.

### Что важно

Service worker должен быть простым и устойчивым:

- не смешивать туда бизнес-логику;
- не использовать его как место для сложной аналитики;
- сделать его узким по ответственности.

## 3. Подписка на push

После разрешения пользователя приложение должно:

1. зарегистрировать service worker;
2. получить `PushSubscription` через `registration.pushManager.subscribe(...)`;
3. отправить подписку на сервер;
4. сохранить подписку в БД.

Разрешение на уведомления нужно запрашивать только по явному действию пользователя.

Не стоит открывать нативный запрос permission сразу при первом заходе.

## 4. Хранение подписок в Supabase

Нужна отдельная таблица.

### Рекомендуемая таблица `push_subscriptions`

- `id`
- `user_id`
- `endpoint`
- `p256dh`
- `auth`
- `device_label`
- `user_agent`
- `platform`
- `is_active`
- `last_seen_at`
- `created_at`
- `updated_at`

### Зачем отдельная таблица

- у одного пользователя может быть несколько устройств;
- подписки могут устаревать;
- подписка должна отключаться отдельно от пользователя;
- нужен контроль активности по конкретному устройству.

### Логика таблицы

- одна строка = одна push-подписка одного устройства;
- если устройство переподписалось, старая подписка отключается;
- если push-сервис вернул invalid subscription, подписка помечается `is_active = false`.

## 5. Настройки уведомлений

Нужен пользовательский слой настроек.

### Рекомендуемая таблица `notification_preferences`

- `user_id`
- `appointments_enabled`
- `feedback_enabled`
- `recommendations_enabled`
- `marketing_enabled`
- `quiet_hours_from`
- `quiet_hours_to`
- `created_at`
- `updated_at`

Если не хочется усложнять сразу, можно начать с более простой версии:

- `user_id`
- `push_enabled`
- `appointments_enabled`
- `recommendations_enabled`

## 6. События уведомлений

Для этого проекта логичны такие события:

- новая запись клиента;
- отмена записи;
- перенос записи;
- напоминание о завтрашних записях;
- новый отзыв;
- готова AI-рекомендация;
- критичное падение рейтинга;
- большое количество негативных отзывов за короткий период.

## Что делать первым MVP

Для первого этапа я бы выбрал только 2-3 события:

- готова AI-рекомендация;
- завтра есть записи;
- пришел новый отзыв.

Это дает ценность сразу и не распыляет систему.

## 7. Backend-слой отправки push

Нужен серверный модуль, который:

- получает список активных подписок пользователя;
- формирует payload;
- отправляет web push;
- ловит ошибки доставки;
- деактивирует битые подписки.

### Что понадобится

- VAPID-ключи;
- серверная функция отправки уведомлений;
- API или cron/worker для запуска отправки.

## 8. История уведомлений

Это не строго обязательно для MVP, но полезно.

### Рекомендуемая таблица `notification_events`

- `id`
- `user_id`
- `type`
- `title`
- `body`
- `payload`
- `status`
- `sent_at`
- `clicked_at`
- `created_at`

### Зачем она нужна

- аудит;
- отладка;
- история отправок;
- понимание, что и когда было отправлено;
- диагностика жалоб вида "мне не пришло уведомление".

## Рекомендуемая структура данных

Минимальный устойчивый набор:

### Таблица `push_subscriptions`

Для хранения подписок устройств.

### Таблица `notification_preferences`

Для хранения пользовательских настроек.

### Таблица `notification_events`

Для лога и диагностики.

MVP можно начать даже без `notification_events`, но в долгую лучше добавить.

## UX-рекомендации

### Как просить разрешение правильно

Не нужно запрашивать permission сразу.

Правильнее:

1. показать короткую выгоду;
2. дать понятную кнопку;
3. только после клика открывать системный prompt.

Примеры хороших формулировок:

- "Включить уведомления о новых записях"
- "Получать напоминания и обновления"
- "Уведомлять, когда рекомендации готовы"

### Что важно для iPhone

Нужно отдельно подсказать пользователю:

- что приложение можно добавить на главный экран;
- что уведомления работают лучше именно в установленной версии.

То есть нужен UX-поток не только "включить push", но и "установить приложение".

## Что не стоит делать сразу

Я бы не делал в первом релизе:

- слишком много типов уведомлений;
- сложные сегменты и маркетинговые рассылки;
- частые пуши без настройки quiet hours;
- смешивание push с email и Telegram в один слой без разделения ответственности.

Сначала нужен устойчивый push-канал.

## Риски

### 1. Слишком ранний permission prompt

Если показать системный prompt без контекста, пользователь откажет, и потом вернуть его сложнее.

### 2. Отсутствие деактивации битых подписок

Если не чистить невалидные subscriptions, серверный слой быстро зашумится.

### 3. Push без продуктовой логики

Если уведомления будут неценными или слишком частыми, пользователь просто отключит их.

### 4. Игнорирование Home Screen сценария

Если строить UX только под обычную браузерную вкладку, iPhone-опыт будет хуже, чем ожидается.

## Финальный вердикт

Для этого проекта push-уведомления делать стоит.

Правильный путь:

- строить решение как `PWA + Home Screen web app`;
- хранить подписки отдельно;
- отправку делать с сервера;
- начинать с небольшого MVP;
- потом уже наращивать события, историю и настройки.

Если делать это поэтапно и без лишнего усложнения, приложение сможет давать опыт, близкий к обычному мобильному приложению.
