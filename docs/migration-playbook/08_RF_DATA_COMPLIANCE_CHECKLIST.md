# 08. RF Data Compliance Checklist

## Зачем этот документ

Этот файл нужен как практический checklist под `mbooking` с учетом того, что проект работает с персональными данными клиентов в РФ.

Это не юридическое заключение.  
Это рабочий технический и организационный список того, что нужно проверить перед миграцией и после нее.

## 1. Почему это важно именно для `mbooking`

В проекте уже есть персональные данные:

- имя клиента
- телефон клиента
- записи на прием
- история посещений
- скидки клиента
- feedback, если он связан с записью или клиентом
- client portal invites и client portal profiles
- push subscriptions

То есть проект явно работает с ПДн клиентов.

## 2. Базовое правило локализации

Практический смысл для проекта такой:

- основная запись, систематизация, накопление, хранение, уточнение и извлечение персональных данных граждан РФ должны идти с использованием баз данных, находящихся на территории РФ

Для вас это значит:

- production database лучше держать в РФ
- backend, который работает с production ПДн, тоже лучше держать в РФ
- резервные сценарии и внешние интеграции тоже нужно проверить, чтобы не оказалось скрытого вывода ПДн за пределы РФ

## 3. Что в `mbooking` нужно считать зонами с ПДн

Критичные сущности:

- `clients`
- `appointments`
- `client_discounts`
- `feedback_tokens`
- `feedback_responses`
- `client_portal_invites`
- `client_portal_profiles`
- `client_portal_links`
- `push_subscriptions`

Также ПДн могут утекать через:

- логи backend
- error monitoring
- analytics
- email-письма
- push payloads
- backup files

## 4. Что нужно локализовать в РФ в первую очередь

Минимальный обязательный контур:

- production PostgreSQL
- production backend API
- резервные копии production БД
- файлы с экспортами данных
- cron/jobs, которые работают с ПДн

Практически для вас это означает:

- `Timeweb Cloud Server` в РФ подходит как базовая точка размещения
- `Vercel + Supabase` уже заметно хуже подходят под задачу локализации

## 5. Что проверить по внешним сервисам

### AI

Проверить:

- отправляются ли в AI-провайдер реальные тексты отзывов
- отправляются ли туда телефоны, имена или иные идентификаторы
- можно ли анонимизировать payload перед отправкой

Практическое правило:

- в LLM не отправлять телефон, email, invite token, raw client identity

### Push

Проверить:

- что именно уходит в push payload
- нет ли там лишних ПДн

Практическое правило:

- push должен содержать минимум данных
- лучше не слать полный телефон, email или чувствительные детали

### Email

Если позже будет email-рассылка:

- проверить, где расположен email provider
- проверить, какие именно данные туда передаются

### Monitoring / logs

Проверить:

- не пишутся ли телефоны и email в логи
- не отправляются ли stack traces с ПДн во внешние системы

## 6. Что проверить по текущему проекту перед миграцией

Нужно сделать inventory:

- какие таблицы содержат ПДн
- какие route handlers читают ПДн
- какие cron/jobs трогают ПДн
- какие внешние сервисы получают payload с ПДн
- где хранятся бэкапы
- где хранятся временные SQL export/import файлы

## 7. Чеклист перед production-переездом

### Инфраструктура

- production DB размещена в РФ
- backend размещен в РФ
- backup storage размещено в РФ или отдельно проверено на допустимость
- SSH доступ защищен
- env-файлы не содержат лишних копий ПДн

### Приложение

- frontend больше не зависит от зарубежного backend для основной работы с ПДн
- auth больше не зависит от Supabase Auth
- основные CRUD по клиентам и записям идут через backend в РФ
- reminders/jobs выполняются на сервере в РФ

### Данные

- перенос данных выполнен без потери UUID и связей
- старые дампы не лежат бесконтрольно на локальных машинах
- после миграции есть понятный порядок удаления временных export/import файлов

## 8. Что проверить после production-переезда

### Smoke check

- клиент создается
- запись создается
- скидка назначается
- feedback работает
- клиентский кабинет работает
- push работает

### Compliance check

- production writes больше не идут в Supabase
- актуальная база ПДн находится в РФ
- логи не содержат лишних ПДн
- резервные копии делаются на новом контуре

## 9. Что еще нужно сделать организационно

Технический переезд сам по себе не закрывает все вопросы.

Обычно отдельно нужно проверить:

- уведомление в Роскомнадзор об обработке ПДн, если оно требуется
- наличие политики обработки персональных данных
- описание мер защиты ПДн
- актуальность пользовательских документов и согласий
- порядок работы с подрядчиками и сервисами, которым передаются данные

Это уже не чисто dev-задача, но игнорировать это нельзя.

## 10. Практический вывод для `mbooking`

Для вашего проекта переход:

- с `Vercel`
- с `Supabase`
- на `Timeweb + свой backend + PostgreSQL`

выглядит не просто как техническое улучшение, а как нормальный путь под локализацию ПДн в РФ.

С технической стороны самый правильный путь такой:

1. Сначала переписать backend на `NestJS`.
2. Перевести auth и data layer с Supabase.
3. Поднять production в РФ.
4. Проверить, что production writes идут только в новый контур.
5. Только потом окончательно отказаться от старой платформы.
