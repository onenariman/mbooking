# MBOOKING PRODUCT SPEC (CLIENT FLOW FIRST)

## 0. ГЛАВНАЯ ЦЕЛЬ ПРОДУКТА

Система должна не просто хранить записи, а **увеличивать доходимость клиентов и повторные визиты**.

Ключевые метрики:

1. Конверсия в запись
2. Снижение no-show
3. Повторные визиты (rebooking)

---

## 1. ОСНОВНОЙ ПРИНЦИП РАЗРАБОТКИ

1. Сначала реализуется клиентский путь (client flow)
2. Только потом — внутренняя CRM
3. Каждая функция должна отвечать на вопрос:
   → "Как это влияет на деньги или поведение клиента?"

---

## 2. CORE FLOW (ОБЯЗАТЕЛЬНО РЕАЛИЗОВАТЬ СНАЧАЛА)

### 2.1 Онлайн-запись (Public Booking Page)

Задача:
Создать публичную страницу записи по ссылке.

Функционал:

1. Выбор услуги
2. Выбор даты и времени
3. Выбор специалиста (если есть)
4. Создание записи
5. После записи → редирект на страницу визита

URL пример:
`/booking/:companyId`

---

### 2.2 Страница визита (Visit Page) — КЛЮЧЕВАЯ СТРАНИЦА

Задача:
Создать страницу, которая открывается по ссылке из сообщения/QR.

URL:
`/visit/:visitId`

Функционал:

1. Отображение:
   - дата и время
   - услуга
   - специалист

2. Кнопки:
   - Подтвердить (confirm)
   - Перенести (reschedule)
   - Отменить (cancel)

3. Изменение статуса визита:
   - pending
   - confirmed
   - cancelled
   - no_show (позже)
   - arrived (позже)

---

### 2.3 Генерация ссылки

После создания записи система должна:

1. Сгенерировать уникальную ссылку:
   `/visit/:visitId`
2. Позволить:
   - скопировать ссылку
   - показать QR-код

---

## 3. SELF-SERVICE (ВМЕСТО ЗАВИСИМОСТИ ОТ СООБЩЕНИЙ)

### 3.1 Все действия через страницу визита

Пользователь должен:

- подтвердить
- отменить
- перенести

БЕЗ обязательных сообщений (WhatsApp/SMS)

---

## 4. PWA (ОСНОВА УВЕДОМЛЕНИЙ)

### 4.1 Добавить поддержку PWA

Обязательно:

1. manifest.json
2. service worker
3. иконки
4. HTTPS

---

### 4.2 Запрос разрешения на уведомления

НЕ показывать сразу.

Показать после:

- открытия страницы визита
  ИЛИ
- действия (например confirm)

Текст:
"Хотите получить напоминание о визите?"

---

### 4.3 Логика push-уведомлений

Если пользователь дал разрешение:

Отправлять:

1. за 24 часа
2. за 2 часа

Push должен содержать:

- дату
- кнопку "Открыть запись"

---

## 5. iPHONE UX (ОЧЕНЬ ВАЖНО)

### 5.1 Проблема:

Push работает только если сайт добавлен на экран

---

### 5.2 Решение:

Показать блок:

"Чтобы получать напоминания:

1. Нажмите Поделиться
2. Выберите 'На экран Домой'"

---

### 5.3 UI-компонент:

Создать компонент:
`InstallPromptIOS`

Показывать:

- только для iOS
- не сразу
- по кнопке "Получать напоминания"

---

## 6. ANDROID INSTALL PROMPT

Реализовать:

- beforeinstallprompt event
- кнопку "Добавить на экран"

---

## 7. STORIES UI (УЛУЧШЕНИЕ UX, НЕ CORE)

### 7.1 Создать компонент сторис

Компоненты:

- StoriesBar
- StoryViewer

---

### 7.2 Использовать сторис ТОЛЬКО для действий

Примеры сторис:

1. "Ваша запись"
   → открыть визит

2. "Не забудьте визит"
   → включить уведомления

3. "Добавьте на экран"
   → показать инструкцию

4. "Записаться снова"
   → переход к booking

---

## 8. REBOOKING (ПОВТОРНЫЕ ВИЗИТЫ)

Минимальный функционал:

1. Кнопка:
   "Записаться снова"

2. Авто-предложение:
   через N дней (backend позже)

---

## 9. FALLBACK (ЕСЛИ НЕТ PUSH)

Система должна работать без:

- WhatsApp
- Telegram
- SMS

Push = бонус, а не зависимость

---

## 10. НЕ РЕАЛИЗОВЫВАТЬ СЕЙЧАС

1. POS / касса
2. склад
3. payroll
4. marketplace
5. сложная CRM

---

## 11. ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ

Frontend:

- React
- Tailwind

Routing:

- /booking/:companyId
- /visit/:visitId

Backend:

- REST API или RPC

---

## 12. ГЛАВНОЕ ПРАВИЛО

Любая новая функция должна проверяться:

→ Увеличивает ли она:

- запись
- доходимость
- повторные визиты

Если нет — не делать.
